# Freiland Covenant — консенсус-версия форсированной продажи (nv3 софтфорк)

Статус: **DESIGN** (2026-07-23). Реализация — многосессионный консенсус-проект в
`/root/fc-nv3` (C++), обкатка на выделенном регтесте → отдельный сигнет → активация.
Заменяет релейный MVP форс-выкупа (см. `freiland-spec.md` §5-6, §7 «ковенант-версия»).

---

## 0. Зачем (и почему релейной версии мало)

Релейный MVP держит форс-продажу на **подписи владельца** над ranged-дескриптором:
подпись истекает (`nExpireTime`/lock_height), поэтому её надо перевыставлять (маячок
`maybeResignLand`), а значит нужен онлайн-владелец **и** релей, хостящий стоячий оффер.
Отсюда три болячки, разобранные с пользователем:
1. **Офлайн-владелец** → оффер истекает → имя временно НЕ выкупаемо.
2. **Прокрутка залога**: залог продавца выходит, залог покупателя входит отдельной
   монетой ⇒ покупателю нужно **2V** ликвидности (цена + свой свежий залог), хотя
   экономически расход — только V.
3. Цена сползает за тающим залогом лишь на перевыставлениях (шагами, и только пока
   владелец онлайн), а не непрерывно.

Ковенант убирает всё это: условия форс-продажи переносятся из подписи **в сам выход**,
и консенсус авторизует выкуп без подписи владельца.

## 1. Что уже есть в nv3 (фундамент)

- **`OP_CAT`** включён (`script.h:150`) — полноценный ковенант-примитив на уровне скрипта.
- **Ranged-бандлы ЗАШИТЫ в консенсус** (`CRangedBundle` в `primitives/transaction.h`,
  валидация в `consensus/tx_verify.cpp:302-334`): нода сама проверяет payout-скрипт,
  цену (`priceNum/priceDen`, округление в пользу мейкера), границы фила (`minFill/maxFill`)
  и что `fill = givePV(lock_height) − change`. **Механика форс-продажи уже в консенсусе.**
- **Present value активов в консенсусе** (`asset_pv` → `TimeAdjustValueForwardK`,
  `tx_verify.cpp:261-265`) — демерредж-таяние считает нода.
- **Deployment-флаги** (`consensus/params.h`: `PROTOCOL_CLEANUP`, `SIZE_EXPANSION`) —
  механика активации новых правил по высоте/versionbits на месте.

Единственное, чего не хватает: give-вход ranged-оффера авторизуется **CHECKSIG-подписью
мейкера** (SIGHASH_BUNDLE, `interpreter.cpp:1978`). Её и убираем.

## 2. Ключевая идея — слить NFT и залог в ОДИН ковенант-выход

Сейчас имя = NFT-токен (объект 1) + отдельный FRC-залог (объект 2). Это и рождает
«прокрутку». В ковенанте — **один выход**:

```
Harberger-output = {
    value:      депозит в кариях (тает демерреджем = рента),
    assetTag:   land-NFT (1 токен, tag коммитит имя),
    scriptPubKey: HARBERGER-ковенант, коммитящий (ownerKey, ...)
}
```

Монета одновременно и токен-имя, и залог. **Цена = present value этой же монеты** — то
самое «одно число», из-за которого пользователь трижды переспрашивал, теперь буквально
одна монета. Отсюда:
- **Форс-выкуп: покупатель платит V старому владельцу, монета-имя переезжает к покупателю**
  как его залог. **Нужна только V, не 2V.** Прокрутки нет — монета остаётся на месте,
  меняется лишь ownerKey в ковенанте.
- **Цена сползает автоматически**: V выкупа = `asset_pv(value, tip − refheight)`, считает
  консенсус на каждом блоке. Тающий залог = падающая цена, без перевыставлений и без
  онлайн-владельца.

## 3. Ковенант-выход: что коммитит scriptPubKey

Witness-программа нового типа (напр. witness v2 «HRBG» или тег в extension-выходе, как
assetTag уже ездит — `tx_verify.cpp:317` `GetWitnessBase`), коммитящая:
- **`ownerKey`** — куда идёт V при форс-выкупе (адрес выплаты владельцу);
- (опц.) **`floorV`** — минимальная цена, чтобы демерредж не увёл в 0 (dust-пол по Гезеллю,
  ср. `LAND_MIN_V`); ниже — имя «протухло», свободно к захвату за floorV.
Сумма выхода (`value`) = залог; его `asset_pv` = текущая цена. `V` отдельно НЕ хранится —
он ВЫВОДИТСЯ из value (в этом суть слияния).

## 4. Пути траты (консенсус, расширение ranged-логики §2 tx_verify)

**(A) Форс-выкуп — ЛЮБОЙ, без подписи владельца.** Спенд Harberger-выхода валиден, если tx:
1. платит `V = asset_pv(value, tip−refheight)` FRC на `ownerKey` (проверка как ranged-payout,
   но payoutScript/цена берутся из КОВЕНАНТА выхода, не из подписанного дескриптора);
2. создаёт выход-преемник: тот же land-NFT в новом Harberger-ковенанте с `ownerKey' = buyerKey`
   (и свежим value ≥ floorV — покупатель заявляет свою V');
3. НЕ требует подписи старого владельца — авторизует ковенант (терминальное правило в
   `VerifyScript`/`tx_verify`: HARBERGER-вход + корректный payout+преемник ⇒ OK).

**(B) Переоценка/долив/вывод — владелец, С подписью.** Обычный спенд под `ownerKey` (CHECKSIG):
меняет value (долить/снизить), пересобирает ковенант с новым floorV, либо выводит имя из-под
ковенанта (отказ). Здесь подпись нужна и норм — это действия САМОГО владельца.

**(C) Резолвер имя→адрес** — как в MVP: witness-extension выхода или FRAN-компаньон; читается
индексаторами, консенсусом игнорируется.

## 5. Демерредж и «протухание»

- Цена выкупа = `asset_pv(value, dist)` — падает сама. Долив (путь B) поднимает value → цену.
- Пол `floorV` (по Гезеллю — технический dust, [[one-unit-coin-freeze]]): ниже него имя
  считается заброшенным и берётся за floorV кем угодно; резолв гаснет. Развязка «вход-порог
  vs протухание» решается тем, что floorV коммитится в выход per-имя, а не глобально.
- 1-юнитная заморозка ([[one-unit-coin-freeze]]) НЕ грозит: залог — FRC-хост-монета, номинал
  большой; NFT-токен едет 0-юнитным extension'ом, не участвует в PV-балансе.

## 6. Активация (софтфорк)

Новое правило = deployment-флаг рядом с `PROTOCOL_CLEANUP`/`SIZE_EXPANSION` (или versionbits
`DEPLOYMENT_*`). До активации HARBERGER-выход = anyone-can-spend по старым правилам (soft-fork-
совместимо: старые ноды видят witness-выход как обычный, новые — энфорсят ковенант). Обкатка:
выделенный регтест → новый сигнет → активация по высоте.

## 7. Порядок сборки (каждая фаза — своя цепочка + тесты, консенсус НЕ трогаем без обкатки)

1. **Спека скрипт-формата** HARBERGER-выхода (байты witness-программы/тега, что коммитит).
   Юнит-тесты сериализации в JS-модели (`apps/web/test/`) + C++ (`asset_tests`).
2. **Консенсус (регтест-only флаг):** распознавание Harberger-выхода + путь (A) в `tx_verify`
   (payout=asset_pv из ковенанта, преемник-выход, без подписи). Функциональные тесты на
   форс-выкуп офлайн-владельца, авто-сползание цены, floorV-протухание.
3. **Путь (B)** владельца (переоценка/долив/вывод под подписью) + консенсус-проверки.
4. **Кошелёк:** строить/распознавать Harberger-выходы (замена релейного registerName/buyName/
   revalueName на ковенант-транзакции); релей → чистый индексатор (без стоячих офферов, без
   maybeResignLand). E2E на регтесте: выкуп при офлайн-владельце, покупателю хватает V.
5. **Deployment-флаг + активация** на новом сигнете; миграция/сосуществование с релейными именами.
6. **Аудит** (консенсус-код — высшая ставка; отдельный adversarial-проход, ср.
   [[security-audit-2026-07-16]]) → активация.

## 8. Риски / открытые вопросы

- **Консенсус-баг = раскол сети / потеря средств.** Только через регтест→сигнет, ревью, аудит.
  Никаких правок `tx_verify` на живом nv3-сигнете до полной обкатки на отдельной цепи.
- Формат ковенанта: чистый OP_CAT-скрипт (без софтфорка, но хрупко и не тянет PV-арифметику)
  vs новый консенсус-тип (софтфорк, чисто, тянет asset_pv) — **выбран консенсус-тип** (§4),
  OP_CAT-путь отвергнут как непрактичный для демерредж-арифметики.
- Реоценка value вверх при живом ковенанте: как атомарно долить, не открыв окно дешёвого
  выкупа между спендами (аналог гонки из релейной 5c) — проверить на регтесте.
- Сосуществование релейных (объект-1+2) и ковенантных (слитых) имён в одном реестре/UI.
