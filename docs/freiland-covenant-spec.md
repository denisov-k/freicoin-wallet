# Freiland Covenant — консенсус-версия форсированной продажи (nv3 софтфорк)

Статус: **DESIGN** (2026-07-23). Реализация — многосессионный консенсус-проект в
`/root/fc-nv3` (C++), обкатка на выделенном регтесте → отдельный сигнет → активация.
Заменяет релейный MVP форс-выкупа (см. `freiland-spec.md` §5-6, §7 «ковенант-версия»).

---

## 0. Зачем (и почему релейной версии мало)

Релейный MVP держит форс-продажу на **подписи владельца** над ranged-дескриптором:
подпись истекает (`nExpireTime`/lock_height), поэтому её надо перевыставлять (маячок
`maybeResignLand`), а значит нужен онлайн-владелец **и** релей, хостящий стоячий оффер.
Отсюда три болячки, разобранные с пользователем:
1. **Офлайн-владелец** → оффер истекает → имя временно НЕ выкупаемо.
2. **Прокрутка залога**: залог продавца выходит, залог покупателя входит отдельной
   монетой ⇒ покупателю нужно **2V** ликвидности (цена + свой свежий залог), хотя
   экономически расход — только V.
3. Цена сползает за тающим залогом лишь на перевыставлениях (шагами, и только пока
   владелец онлайн), а не непрерывно.

Ковенант убирает всё это: условия форс-продажи переносятся из подписи **в сам выход**,
и консенсус авторизует выкуп без подписи владельца.

## 1. Что уже есть в nv3 (фундамент)

- **`OP_CAT`** включён (`script.h:150`) — полноценный ковенант-примитив на уровне скрипта.
- **Ranged-бандлы ЗАШИТЫ в консенсус** (`CRangedBundle` в `primitives/transaction.h`,
  валидация в `consensus/tx_verify.cpp:302-334`): нода сама проверяет payout-скрипт,
  цену (`priceNum/priceDen`, округление в пользу мейкера), границы фила (`minFill/maxFill`)
  и что `fill = givePV(lock_height) − change`. **Механика форс-продажи уже в консенсусе.**
- **Present value активов в консенсусе** (`asset_pv` → `TimeAdjustValueForwardK`,
  `tx_verify.cpp:261-265`) — демерредж-таяние считает нода.
- **Deployment-флаги** (`consensus/params.h`: `PROTOCOL_CLEANUP`, `SIZE_EXPANSION`) —
  механика активации новых правил по высоте/versionbits на месте.

Единственное, чего не хватает: give-вход ranged-оффера авторизуется **CHECKSIG-подписью
мейкера** (SIGHASH_BUNDLE, `interpreter.cpp:1978`). Её и убираем.

## 2. Ключевая идея — залог=цена в ОДНОЙ монете, и «переиспользование, не прокрутка»

Цель: форс-выкуп = «покупатель платит V старому владельцу, монета-залог **переезжает** к
покупателю» (не уничтожается+пересоздаётся). Тогда покупателю нужна только **V, не 2V**
(прокрутки нет), а цена сползает сама: `V = asset_pv(value, tip − refheight)` считает
консенсус на каждом блоке — тающий залог = падающая цена, без перевыставлений и офлайн.

### ⚠ Ограничение, вскрытое на стадии 1: у выхода ОДИН assetTag

Нельзя «слить NFT-токен и FRC-залог в один выход» — выход несёт один актив и значение в его
единицах. Токен-имя (константный, не тает) и FRC-залог (тает демерреджем) — **разные активы**.
Значит либо залог — host-FRC, а имя как-то отдельно, либо имя — актив, а залог отдельно. Отсюда
развилка (§2a), которую надо решить ДО формата — она консенсусная, разворота не будет.

### §2a. Развилка: чем обеспечена уникальность имени и где живёт тающий залог

**Вариант A — host-FRC-ковенант + консенсус-реестр имён (РЕКОМЕНДУЮ).**
Harberger-выход = ОДНА host-FRC-монета:
```
{ value: залог (host FRC, тает k=20 = рента), assetTag: null,
  scriptPubKey: HARBERGER-ковенант, коммитящий (ownerKey, nameHash) }
```
Цена = `asset_pv(value)` — тает сама. Имя-токена НЕТ вовсе. Уникальность — новый консенсус-
реестр `nameHash → живой outpoint` (зеркало существующего asset-registry; при форс-выкупе/
переоценке запись переезжает на новый выход). Форс-выкуп и переоценка — спенд ОДНОГО выхода,
монета переиспользуется (owner меняется, value несёт). Truly «одна монета», без биндинга.
_Плюс:_ чистейший выход, залог — понятный host-демерредж, нет двух-выходного ковенанта.
_Минус:_ новый (но простой, map-like) реестр имён в консенсусе.

**Вариант B — имя как asset-токен + связанный FRC-залог (два выхода).**
Имя = nv3-актив (tag=hash160(имени), уникальность БЕСПЛАТНО из asset-registry). Ковенант на
ОТДЕЛЬНОМ host-FRC-залоговом выходе, консенсус-биндингом связанном с NFT: спенд любого требует
спенда обоих + правил форс-выкупа. Залог при выкупе переиспользуется (переезжает к покупателю).
_Плюс:_ уникальность даром, extension-output/token-код (C++ и JS) уже есть и оттестирован.
_Минус:_ атомарный биндинг ДВУХ выходов — самая хрупкая, ошибкоёмкая часть; сложнее реестра.

**Рекомендация: A.** Новый map-реестр имён проще и безопаснее двух-выходного атомарного
ковенанта; выход честно один (та самая «одна монета»), залог — обычный host-FRC-демерредж,
без специальной token-арифметики. Токен-NFT в ковенант-мире просто не нужен. Формат §3 и
пути §4 ниже расписаны под A; при выборе B переписываются (два выхода + биндинг-коммитмент).

## 3. Ковенант-выход: что коммитит scriptPubKey (под вариант A)

**ИСПРАВЛЕНО фазой 2 (см. §7a): не v0-base, а НОВАЯ witness-версия.** v0-base (`0014{hash}`)
энфорсится старой нодой (MAST-корень) ⇒ форс-выкуп без подписи там = ХАРД-форк. Неизвестная
witness-версия — anyone-can-spend для старых нод ⇒ энфорс новых только ужесточает = СОФТфорк.
HRBG = **witness-версия 2** (внешний байт OP_1=0x51), extension-суффикс несёт ковенант, OP_3-маркер:
```
scriptPubKey = 51 20{nameHash(32)} 14{ownerHash160(20)} 08{floorV(8 LE)} OP_3
               ^witver-2  ^программа=nameHash  ^суффикс: owner + floorV      ^HRBG-маркер
```
Коммитит:
- **`nameHash`** (32, в программе witver-2) — sha256 имени; ключ консенсус-реестра `nameHash→outpoint`.
- **`ownerHash160`** (20, суффикс) — куда идёт V при форс-выкупе (`0014{owner}`).
- **`floorV`** (8, кария, суффикс) — dust-пол по Гезеллю ([[one-unit-coin-freeze]]); ниже — «протухло»,
  берётся за floorV кем угодно. Per-имя ⇒ развязка «вход-порог vs протухание».

Сумма выхода (`value`) = залог (host FRC; `DeriveAssetTag`: 28-байт-суффикс → assetTag null);
`asset_pv(value, dist)` = текущая цена V (отдельного поля V нет — выводится из value). До активации
софтфорка HRBG-выход старым нодам = **anyone-can-spend** (witver 2 неизвестна) — soft-fork-корректно;
**поэтому HRBG-выходы чеканить ТОЛЬКО после активации** (до неё их кто угодно потратит).

## 4. Пути траты (консенсус, расширение ranged-логики §2 tx_verify)

**(A) Форс-выкуп — ЛЮБОЙ, без подписи владельца.** Экономика выведена (стадия 2): нужны ДВА
правила ВМЕСТЕ, иначе «бесплатный захват». Спенд HRBG-входа `(owner, value=D)` валиден при
`rules & HARBERGER`, если tx:
1. **платит `V = asset_pv(D, tip−refheight)` FRC на `0014{owner}`** (host-выход) — цена продавцу;
2. **создаёт выход-преемник HRBG для того же `nameHash` с `value ≥ V`** (залог «переезжает»,
   не усыхает ниже цены; новый `owner'=buyerKey`, свой `floorV`).
3. НЕ требует подписи старого владельца — авторизует ковенант (терминальное правило в `tx_verify`,
   как ranged-бандл: HRBG-вход + корректные payout+преемник ⇒ OK; иначе Invalid).

**Почему оба правила, а не только (1):** host-FRC фунгибелен, консенсус не отследит «каким входом
оплачен выход». Только правило (1) → покупатель оплатил бы V продавцу из ЕГО ЖЕ залога (HRBG-вход
даёт V), а свой залог-преемник — из своих; продавец получил бы назад лишь свой залог, покупатель
взял бы имя фактически даром (только за свой возвратный залог) = сквоттинг не сдержан. Добавляем
(2) `преемник ≥ V`: тогда host-баланс `D + newCoins ≥ V(продавцу) + (≥V)(преемник) + fee` ⇒
`newCoins ≥ V + fee` — покупатель ОБЯЗАН принести V свежими. Итог: покупателю нужно V+fee
ликвидности, чистый расход = fee + будущая рента (его залог возвратен); продавцу приходит V
(его самооценка). Ровно Харбергер, и «покупателю хватает V» держится.

**Развязка «одна монета» vs правило-2:** залог-преемник ≥ V — это и есть «переезд залога к
покупателю» из §2 (буквального слияния активов нет, но экономически монета-залог непрерывна).

**(B) Переоценка/долив/вывод — владелец, С подписью.** Обычный спенд под `ownerKey` (CHECKSIG):
меняет value (долить/снизить), пересобирает ковенант с новым floorV, либо выводит имя из-под
ковенанта (отказ). Здесь подпись нужна и норм — это действия САМОГО владельца.

**(C) Резолвер имя→адрес** — как в MVP: witness-extension выхода или FRAN-компаньон; читается
индексаторами, консенсусом игнорируется.

## 5. Демерредж и «протухание»

- Цена выкупа = `asset_pv(value, dist)` — падает сама. Долив (путь B) поднимает value → цену.
- Пол `floorV` (по Гезеллю — технический dust, [[one-unit-coin-freeze]]): ниже него имя
  считается заброшенным и берётся за floorV кем угодно; резолв гаснет. Развязка «вход-порог
  vs протухание» решается тем, что floorV коммитится в выход per-имя, а не глобально.
- 1-юнитная заморозка ([[one-unit-coin-freeze]]) НЕ грозит: залог — FRC-хост-монета, номинал
  большой; NFT-токен едет 0-юнитным extension'ом, не участвует в PV-балансе.

## 6. Активация (софтфорк)

Новое правило = deployment-флаг рядом с `PROTOCOL_CLEANUP`/`SIZE_EXPANSION` (или versionbits
`DEPLOYMENT_*`). До активации HARBERGER-выход = anyone-can-spend по старым правилам (soft-fork-
совместимо: старые ноды видят witness-выход как обычный, новые — энфорсят ковенант). Обкатка:
выделенный регтест → новый сигнет → активация по высоте.

## 8. Риски / открытые вопросы

- **Консенсус-баг = раскол сети / потеря средств.** Только через регтест→сигнет, ревью, аудит.
  Никаких правок `tx_verify` на живом nv3-сигнете до полной обкатки на отдельной цепи.
- Формат ковенанта: чистый OP_CAT-скрипт (без софтфорка, но хрупко и не тянет PV-арифметику)
  vs новый консенсус-тип (софтфорк, чисто, тянет asset_pv) — **выбран консенсус-тип** (§4),
  OP_CAT-путь отвергнут как непрактичный для демерредж-арифметики.
- Реоценка value вверх при живом ковенанте: как атомарно долить, не открыв окно дешёвого
  выкупа между спендами (аналог гонки из релейной 5c) — проверить на регтесте.
- Сосуществование релейных (объект-1+2) и ковенантных (слитых) имён в одном реестре/UI.
