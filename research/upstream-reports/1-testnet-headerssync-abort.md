# testnet: node aborts during IBD — CTestNetParams is missing m_headers_sync_params

## Summary

A fresh `freicoind -testnet` (rebase-31 / v31 base) crashes with SIGABRT as soon as the first
peer triggers the low-work headers presync path:

```
freicoind: ./headerssync.cpp:44: HeadersSyncState::HeadersSyncState(NodeId, const Consensus::Params&,
const HeadersSyncParams&, const CBlockIndex&, const arith_uint256&):
Assertion `params.commitment_period > 0' failed.
```

## Cause

`CTestNetParams` in `src/kernel/chainparams.cpp` never initializes `m_headers_sync_params`,
so `commitment_period` stays value-initialized to 0. Main, signet and regtest all set it
(641/15218, 620/15724 and 275/7017 respectively); testnet is the only chain type without it.

The assert fires on any headers sync whose advertised chain work is below `nMinimumChainWork`
— which is every initial sync of today's testnet (tip ≈ 33759, well below the checkpointed
minimum-work value), so the crash is deterministic on first peer connect.

## Repro

1. `freicoind -testnet` with an empty datadir.
2. Wait for the first outbound peer (fixed seeds suffice).
3. Node aborts within seconds of `New outbound-full-relay peer connected`.

Workaround for testing: `-minimumchainwork=0x…01` keeps every received batch above the
anti-DoS threshold so the presync path is never entered (obviously not a fix).

## Suggested fix

Generate testnet parameters with `contrib/devtools/headerssync-params.py` and add them to
`CTestNetParams`, mirroring the other chains, e.g.:

```cpp
        // Generated by headerssync-params.py on <date>.
        m_headers_sync_params = HeadersSyncParams{
            .commitment_period = <generated>,
            .redownload_buffer_size = <generated>,
        };
```

Alternatively (or additionally), the HeadersSyncState constructor could reject a
zero `commitment_period` gracefully instead of asserting, since a misconfigured chain
type otherwise turns into a remote-triggered abort.
